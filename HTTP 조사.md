# HTTP

<hr>

## HTTP란?

> - 웹 브라우저(클라이언트)와 웹 서버가 HTML, 이미지, 영상 등 다양한 데이터를 주고받기 위해 사용하는 통신 규약이다.

### 핵심 특징 및 구성

> - **요청 / 응답 구조** : 클라이언트가 요청을 보내고, 서버가 결과를 응답한다.<br>
> - **비연결성** : 서버가 클라이언트와의 연결을 유지하지 않는다. 따라서 필요 시 쿠키(Cookie)나 세션(Session) 기술을 함께 사용한다. <br>
> - **TCP / UDP 기반** : 기본적으로 TCP를 사용하며, HTTP/3부터는 UDP 기반 프로토콜을 사용한다.

<hr>

### HTTP와 HTTPS의 차이

**보안성 (Security)**

    - HTTP : 일반 텍스트로 데이터를 전송한다. 중간에 데이터가 탈취되거나 변조될 수 있다는 문제점이 있다.
    - HTTPS : SSL / TLS 프로토콜을 사용해 데이터를 암호화한다. 제3자가 데이터를 가로채도 내용을 확인할 수 없다. <br>

**작동 방식**

    - HTTP : 80번 포트를 사용하며, 연결 과정이 단순하다.
    - HTTPS : 443번 포트를 사용하며, 통신 시작 전 인증서 검증 및 암호화 키 교환 과정을 거친다. <br>

**신뢰도 및 표시** <br>

    - HTTP : 최근 브라우저들은 HTTP 사이트를 '안전하지 않음'으로 표시한다.
    - HTTPS : 브라우저 주소창에 자물쇠 아이콘이 표시되며, 안전한 사이트로 인증된다. <br>

**SEO (검색 엔진 최적화)** <br>

    - 검색 엔진은 HTTPS를 적용한 사이트에 가산점을 주어 검색 순위에서 우대한다.

<hr>

### Stateless 구조

> - 서버가 클라이언트의 이전 요청 정보를 저장하지 않는 구조이며, 각 요청은 독립적으로 처리된다. <br>
> - 서버 확장성이 뛰어나 대규모 서비스에 적합하고, 중간에 서버가 바뀌어도 요청 처리가 가능하다는 장점이 있다. <br>
> - 매 요청마다 필요한 상태 정보를 클라이언트가 다시 전송해야 하므로 데이터량이 증가할 수 있다는 단점이 있다.

<hr>

### 클라이언트 - 서버 구조

> - 요청(Request) <-> 응답(Response)으로 동작한다. <br>
> - 사용자가 버튼을 클릭하거나 주소를 입력하는 것 등이 요청(Request)에 해당한다.
> - 데이터를 가지고 있는 서버가 요청을 처리하여 결과를 만들어내는 것이 응답(Response)에 해당한다. <br>

### HTTP 통신 규칙

> - 클라이언트와 서버가 대화하는 방법이다. <br>
> - 약속된 형식으로 요청 / 응답을 주고받는다.

<hr>

### 주요 HTTP 메서드

GET

    - 데이터를 가져오는 메서드로, 서버에서 정보를 조회할 때 사용하며 데이터에 변화를 주지 않는다. <br>

POST

    - 데이터를 생성하는 메서드로, 서버에서 새로운 데이터를 추가할 때 사용한다. <br>

PUT

    - 데이터를 수정하는 메서드로, 기존 데이터를 전체 수정할 때 사용한다. <br>

DELETE

    - 데이터를 삭제하는 메서드로, 데이터를 지울 때 사용한다. <br>

<hr>

### HTTP의 상태 코드

> - 1XX : 요청을 받았고 처리 중인 상태를 의미한다. <br>
> - 2XX : 요청이 정상적으로 처리된 상태를 의미한다. <br>
> - 3XX : 다른 주소로 이동하라는 의미를 가진 상태이다. <br>
> - 4XX : 클라이언트(사용자)의 요청이 잘못된 상태를 의미한다. <br>
> - 5XX : 서버에서 문제가 발생한 상태를 의미한다.

### 핵심 상태 코드

> 2XX <br>
>
> - 200 : 요청이 정상적으로 처리됨 <br>
> - 201 : 새 데이터가 생성됨 <br>
> - 204 : 보낼 데이터가 없음

<br>

> 3XX <br>
>
> - 301 : 영구적으로 주소가 바뀜 <br>
> - 302 : 임시 이동 <br>
> - 304 : 변경된 내용이 없어 캐시를 사용함 <br>

<br>

> 4XX <br>
>
> - 400 : 요청 형식이 잘못됨 <br>
> - 401 : 인증이 필요함 <br>
> - 403 : 권한이 없음 <br>
> - 404 : 요청한 내용이 없음 <br>
> - 405 : 요청 방식이 잘못됨 <br>

<br>

> 5XX <br>
>
> - 500 : 서버 내부에서 문제가 발생함 <br>
> - 502 : 서버 간 통신 과정에서 문제가 발생함 <br>
> - 503 : 서버가 과부하 상태이거나 점검 중임 <br>

### HTTP의 요청과 응답 구조

전체적 구조 <br>

    - 시작줄 : Start Line
    - 헤더 : Headers
    - 빈 줄
    - 본문 : Body (선택 사항)

HTTP 요청 구조

    - GET /index.html HTTP/1.1 <= 시작줄
    - Host: example.com <= 헤더
    - User-Agent: Chrome <= 헤더

HTTP 응답 구조

    - HTTP/1.1 200 OK <= 시작줄
    - Content-Type: text/html <= 헤더
    - 빈 줄
    - <html>...</html> <= 바디

<hr>

### 요청과 응답의 차이

| 구분   | 요청                            | 응답                             |
| ------ | ------------------------------- | -------------------------------- |
| 시작줄 | 요청 방식 + URL                 | HTTP 상태 코드 (200, 404 등)     |
| 목적   | 서버에 작업을 요청              | 요청 결과 전달                   |
| 헤더   | 클라이언트 정보, 요청 조건 포함 | 서버 정보, 응답 데이터 정보 포함 |
| 바디   | 서버로 보낼 데이터              | 클라이언트에게 줄 데이터         |

<hr>

### Header와 Body의 역할

<hr>

### Header의 역할

> - 서버 / 클라이언트가 데이터를 어떻게 처리할지에 대한 규칙을 알려주는 역할이다.

주요 역할 4가지

> - 데이터 형식 정의 <br>
> - 인증 / 권한 <br>
> - 동작 방식 제어 <br>
> - 추가 정보 전달 <br>

### Body의 역할

> - 서버가 직접 처리해야 하는 실제 데이터를 의미한다.

사용되는 경우 <br>

    - 회원가입 정보
    - 게시글 내용
    - 파일 업로드

<hr>

### Header / Body를 나눈 이유

1. 역할 분리
   - Header : 처리 방법
   - Body : 처리 대상

2. 유연성 증가
   - 같은 Body라도 Header에 따라 다르게 처리할 수 있다.
3. 성능 & 효율
   - 필요 없는 데이터는 보내지 않아도 된다.
   - 캐시, 압축 등을 제어할 수 있다.
