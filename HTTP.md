# HTTP와 HTTPS

# HTTP

## HTTP란

> - 웹 브라우저(클라이언트)와 웹 서버가 HTML, 이미지, 영상 등 다양한 **데이터를 주고 받기 위해 사용하는 통신 규약**을 의미한다.

## 핵심 특징과 구성

> - 요청 / 응답 구조 : 클라이언트가 요청을 보내고, 서버가 결과를 응답한다.
> - 비연결성 : 서버가 클라이언트와의 연결을 유지하지 않는다. 따라서 필요 시 **쿠키(Cookie)나 세션(Session)** 기술을 함께 사용한다.
> - TCP / UDP 기반 : **기본적으로 TCP**를 사용하고, **HTTP/3부터는 UDP 기반** 프로토콜을 사용한다.

<hr>

## Stateless 구조

> - 서버가 클라이언트의 이전 요청 정보를 저장하지 않는 구조이며, **각 요청은 독립적**으로 처리된다.
> - 서버 확장성이 뛰어나 **대규모 서비스**에 적합하고, 중간에 **서버가 바뀌어도 요청 처리가 가능**하다는 장점을 가지고 있다.
> - 매 요청마다 필요한 상태 정보를 클라이언트가 다시 전송해야 하므로 **데이터량이 증가**할 수 있다는 단점이 있다.

<hr>

## 클라이언트 <ㅡ> 구조

> - 요청(Request) <ㅡ> 응답(Response) 구조로 동작한다.
> - 요청(Request)의 예시로는 **버튼 클릭, 주소 입력 등**이 있다.
> - 응답(Response)는 데이터를 가지고 있는 **서버가 요청을 처리하여 결과를 얻는 것**을 의미한다.

<hr>

## HTTP 통신 규칙

> - 클라이언트와 서버가 대화하는 방법이다.
> - 약속된 형식으로 요청과 응답을 주고받는다.

<br>

## 주요 HTTP 메서드

GET

> - **데이터를 가져오는** 메서드로, 서버에서 정보를 조회할 때 사용하며 **데이터에 변화를 주지 않는다.**
>   POST
> - **데이터를 생성**하는 메서드로, 서버에서 새로운 **데이터를 추가**할 때 사용한다.
>   PUT
> - **데이터를 수정**하는 메서드로, 기존 데이터를 전체 수정할 때 사용한다.
>   DELETE
> - **데이터를 삭제**하는 메서드로, 데이터를 지울 때 사용한다.

<br>

## HTTP의 상태 코드

> - 1XX : 요청을 받았고 **처리중인 상태**를 의미한다.
> - 2XX : 요청이 **정상적으로 처리된 상태**를 의미한다.
> - 3XX : **다른 주소로 이동**하라는 의미를 가진 상태 코드이다.
> - 4XX : 클라이언트(사용자)의 **요청이 잘못된 상태**를 의미한다.
> - 5XX : **서버에서 문제**가 발생한 상태를 의미한다.

## 핵심 상태 코드

### 2XX

> - 200 : 요청이 정상적으로 처리됨
> - 201 : 새 데이터가 생성됨
> - 204 : 보낼 **데이터가 없음**

### 3XX

> - 301 : **영구적**으로 주소가 변경됨
> - 302 : **일시적**으로 주소가 변경됨
> - 304 : 변경된 내용이 없어 **캐시를 사용**함

### 4XX

> - 400 : 요청 형식이 잘못됨
> - 401 : 인증이 필요함
> - 403 : 권한이 없음
> - 404 : 요청한 내용이 없음
> - 405 : 요청 방식이 잘못됨

### 5XX

> - 500 : 서버 **내부에서 문제**가 발생함
> - 502 : 서버 간 **통신 과정**에서 문제가 발생함
> - 503 : 서버가 **과부하 상태**이거나 **점검 중**임

<hr>

## HTTP의 요청과 응답 구조

### 전체적 구조

```
- 시작줄 : Start Line
- 헤더 : Headers
- 빈 줄
- 본문 : Body (선택사항)
```

### HTTP 요청 구조

```
- GET / index.html HTTP/1.1 <= 시작줄
- Host : example.com <= 헤더
- User-Agent: Chrome <= 헤더
```

### HTTP 응답 구조

```
- HTTP/1.1 200 OK <= 시작줄
- Content-Type: Text/html <= 헤더
- 빈 줄
- <html>....</html><= 바디
```

<hr>

## Header와 Body의 역할

### Header의 역할

> - 서버 / 클라이언트가 **데이터를 어떻게 처리할지에 대한 규칙**을 알려주는 역할이다.

주요 역할 4가지

> - 데이터 형식 정의
> - 인증 및 권한
> - 동작 방식 제어
> - 추가 정보 전달

### Body의 역할

> - 서버가 직접 처리해야하는 **실제 데이터**를 의미한다.

사용되는 경우

> - 회원 가입 정보
> - 게시글 내용
> - 파일 업로드

<hr>

## Header 과 Body를 나눈 이유

1. 역할 분리
   - Header : 처리 방법
   - Body : 처리 대상
2. 유연성 증가
   - 같은 Body라도 Header에 따라 다르게 처리될 수 있다.
3. 성능 & 효율
   - 필요 없는 데이터는 보내지 않아도 된다.
   - 캐시, 압축 등을 제어할 수 있다.

# HTTPS

## HTTPS란

> - HTTP에 보안 기술(SSL / TLS)을 더한 **보안 전송 프로토콜**이다.

<hr>

## 핵심 특징과 장점

> - 데이터 암호화 : **통신 내용을 암호화**하여 제3자가 내용을 가로채도 볼 수 없게 암호화 한다.
> - 인증 및 신뢰성 : 인증기관(CA)이 발급한 인증서를 사용해 **접속한 사이트가 진짜인지 확인**하여 피싱 사이트의 위험을 줄인다.
> - 데이터 무결성 : 전송 중 **데이터가 변하지 않음**을 보장한다.
> - 검색 엔진 최적화 (SEO): 구글 등 검색 엔진에서는 **HTTPS 사이트에 가산점을 부여**하여 검색 순위에서 유리하게 작용한다.

<hr>

## HTTPS를 사용해야하는 이유

> - 금융 거래, 로그인 등 개인 정보 입력이 필요한 모든 웹사이트는 필수적으로 사용해야하며, 최신 브라우저(크롬 등)에서는 HTTPS가 아닌 사이트를 "안전하지 않음"으로 표시하여 사용자에게 경고한다.

# HTTPS를

## HTTPS와 HTTPS의 차이점

| 구분                   | HTTP                                                                                    | HTTPS                                                                                         |
| ---------------------- | :-------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------- |
| 보안성(Security)       | 일반 텍스트로 데이터를 송한다. 중간에 데이터가 탈취되거나 변조될 수 있다는 단점이 있다. | SSL / TLS 프로토콜을 사용해 데이터를 암호화한다. 제3자가 데이터를 가로채도 내용을 볼 수 없다. |
| 작동 방식              | 기본적으로 80번 포트를 사용하며, 다른 포트또한 사용 가능하고 연결 과정이 단순하다       | 443번 포트를 사용하며, 통신 시작 전 인증서 검증 및 암호화 키 교환 과정을 거친다.              |
| 신뢰도 및 표시         | 최근 브라우저들은 HTTP 사이트를 "안전하지 않음" 으로 표시한다.                          | 브라우저 주소창에 자물쇠 아이콘이 표시되며, 안전한 사이트로 인증된다.                         |
| SEO (검색 엔진 최적화) | 검색 엔진은 HTTP를 보안 위험 사이트로 간주하여 우선순위에서 밀리게 된다.                | HTTPS를 안전한 사이트로 간주하여 우선순위로 올리게 된다.                                      |
| TLS Handshake          | 해당 없음                                                                               | 통신 시작 전 TLS Handshake를 통해 서버 인증, 암호화 방식 협상, 세션 키 교환을 수행한다.       |
